## System Overview

**One sentence:** An educational full-stack simulator that walks learners through HTTP protocol mechanics — from DNS resolution through TLS handshake to application-layer response — using real Node.js TCP sockets, SSE streaming, and interactive React visualizations.

**Detailed:** The system has two execution planes:

1. **Real plane** — Node.js API routes (`net`, `tls`, `dns` modules) that perform actual network operations and stream events to the browser via `ReadableStream` / `text/event-stream`. Learners see real timings, real certificate metadata, real response bytes.

2. **Virtual plane** — Server-side `setTimeout`-based simulations that emit the same event schema as the real plane. Learners see predictable, jitter-controlled stages without network variability.

Core topics covered: HTTP lifecycle (DNS→TCP→TLS→Request→TTFB→Download), HTTP versions (0.9 / 1.0 / 1.1), Keep-Alive connection reuse, Short Polling, Long Polling, Server-Sent Events (SSE), and Head-of-Line (HOL) blocking.

## Strengths

### Engineering

1. **Dual-mode symmetry.** Virtual and real modes share the same SSE event schema. Frontend rendering code is protocol-agnostic — a well-disciplined architectural choice.

2. **True raw TCP implementation for HTTP lifecycle.** `net.createConnection` → `tls.connect` → `socket.write(rawHttpRequest)` is the real protocol stack, not a `fetch()` wrapper. Learners see actual wire-level behavior.

3. **Keep-alive socket reuse is genuinely demonstrated.** The second request reuses the same `net.Socket` / `tls.TLSSocket` object. DNS, TCP, and TLS stage events are emitted with `status: "reused"` and `duration: 0` — accurately reflecting zero-cost reuse.

4. **Loopback TCP servers for HTTP version comparison.** `net.createServer()` in-process with `listen(0)` (ephemeral port) is a clean way to isolate version-specific wire format without external dependencies.

5. **Session lifecycle in step-mode is correct in principle.** `globalThis.__simSessions` survives Next.js hot-reload. TTL-based cleanup with `clearTimeout` + socket destruction prevents leaks in the happy path.

6. **SSRF protection exists** — the stage route validates resolved IPs against a comprehensive RFC-compliant blocklist (RFC 1918, RFC 5737, RFC 3927, RFC 6598, etc.), including IPv6 ULA and link-local.

7. **Header injection prevention.** RFC 7230 token regex on method/header names and CR/LF check on header values are correct and non-trivial.

8. **Chunked transfer encoding is manually decoded** in every response parser — accurate for the educational context (shows learners that HTTP/1.1 framing is not transparent).

9. **SSE delivery is real-time.** `X-Accel-Buffering: no` disables proxy buffering; `Cache-Control: no-cache, no-transform` prevents caching. Streams close correctly after `controller.close()`.

10. **Type safety throughout.** TypeScript interfaces for requests, sessions, events, and stage states are complete and coherent.

### Educational

1. **Protocol layering is accurate.** DNS → TCP → TLS → HTTP is the correct order; TLS is skipped for HTTP and shown as `status: "skipped"` rather than hidden.

2. **Stage timing granularity.** TTFB (server processing) is separated from download duration — this is the correct distinction and is often conflated in developer tools.

3. **HTTP version wire format differences are shown correctly.** HTTP/0.9 has no status line or headers; HTTP/1.0 lacks `Host`; HTTP/1.1 uses chunked encoding. These are accurate historical facts.

4. **Connection persistence correctly differentiated by version.** 0.9 = stateless close, 1.0 = close by default, 1.1 = keep-alive by default. The server handler enforces this correctly.

### 2. [SECURITY] SSRF Not Enforced in `/api/simulate` (Auto Mode)

`app/api/simulate/route.ts` has NO IP blocklist check. After DNS resolution, it connects to any IP including:
- `127.0.0.1` (loopback) — can reach the Next.js server itself
- `10.x.x.x`, `192.168.x.x` — internal network services
- `169.254.169.254` — AWS EC2 instance metadata endpoint

The stage route (`/api/simulate/stage`) has `isBlockedIp()` — but the primary auto route does not. An attacker with access to the dev/staging server can use this to probe the internal network.

```typescript
// simulate/route.ts — resolveDns() is called but result is never validated:
const ip = await resolveDns(hostname);
// ← No isBlockedIp(ip) check here
socket = await tcpConnect(ip, port);
```

**Impact:** Server-Side Request Forgery. Severity: P1 (high risk if deployed publicly).

### 5. [BUG] Chunked Encoding Decoder Breaks on Chunk Extensions

All four chunked decoders (simulate, stage, long-poll, http-versions) use:

```typescript
const chunkSize = parseInt(remaining.slice(0, crlfIdx), 16);
```

RFC 7230 §4.1 permits chunk extensions: `20;name=value\r\n`. `parseInt("20;name=value", 16)` returns `32` (correct — `parseInt` stops at non-hex), but the slice position `crlfIdx` is still the full extent including the extension. Actually wait — this specific case would *work* because parseInt stops at `;`. However, the decoder then does:

```typescript
remaining = remaining.slice(crlfIdx + 2 + chunkSize + 2);
```

This correctly skips the data. But if a chunk extension exists and the chunk size line is longer than `crlfIdx` + chunksize... actually this *will* parse correctly for standard extensions. However:

The more serious bug: **the final `0\r\n\r\n` terminal chunk** detection in `sendHttpRequestKeepAlive.trySettle()` does a string search:

```typescript
if (body.includes("\r\n0\r\n\r\n") || body.endsWith("0\r\n\r\n"))
```

If the body has an even number of bytes that happen to produce the literal string `0\r\n\r\n` before the actual terminal chunk, this will falsely trigger end-of-response detection mid-stream.

**Impact:** Silent data truncation on specific body content. Severity: P2.

### 6. [SILENT LIE] `savedMs` in Keep-Alive Reuse Always Omits TLS Duration

```typescript
emit(controller, {
  type:    "keep_alive_reuse",
  savedMs: tcpDuration + (isHttps ? 0 : 0),  // ← Bug: both branches are 0
```

The comment says `// TLS ms is in stageData` but `savedMs` silently omits TLS savings when showing connection reuse benefit. A learner sees "saved 14ms" when they actually saved 14ms TCP + 42ms TLS = 56ms. This systematically understates the benefit of keep-alive for HTTPS.

**Impact:** Misleading educational claim. Severity: P2.

### 7. [SILENT LIE] DNS Cache Heuristic Is Unreliable

```typescript
const dnsCached = dnsDuration < 3;
```

- Node.js `dns.lookup()` calls the OS resolver (via libuv), which respects OS-level DNS caching. On Linux, this may cache nothing at all (nscd not running).
- On a fast local network, an uncached DNS round-trip can complete in 1-2ms.
- On a loaded system, even a cached lookup can take >3ms due to thread pool scheduling.

The `cached` flag shown to learners is a guess, not a fact. A learner may believe DNS caching is working when it isn't, or vice versa.

**Impact:** Incorrect mental model formation. Severity: P2.

### 8. [INCORRECT] Request Duration Measurement Is Semantically Wrong

```typescript
const t0 = performance.now();
socket.write(requestRaw, "utf8");
const requestDuration = Math.round(performance.now() - t0);
callbacks.onRequestSent(requestRaw, requestDuration);
```

`socket.write()` is synchronous — it copies data into the OS send buffer. It does **not** wait for the data to traverse the network. The measured duration (typically 0-1ms) reflects kernel buffer write time, not network transmission time. Yet this is displayed as "HTTP Request" duration to learners, implying the request traversed the network instantly.

The real network transit time for the request is embedded in the TTFB measurement (server can't respond until it receives the request). This is never explained.

**Impact:** Fundamental misunderstanding of what "sending a request" means. Severity: P2.

### 9. [INCORRECT] Virtual Timing Values Are Too Optimistic

| Stage | Virtual Range | Real-world P50 |
|-------|--------------|----------------|
| DNS | 8–16ms | 20–80ms (uncached) |
| TCP | 5–13ms | 15–50ms (local CDN), 80–200ms (distant) |
| TLS | 8–18ms | 20–50ms (TLS 1.3), 50–150ms (TLS 1.2) |
| TTFB | variable | 50–500ms (server-dependent) |

A learner using virtual mode will internalize that "DNS takes ~12ms." In production, a cold DNS lookup to a remote resolver takes 30-100ms. This warps the learner's mental model of where latency comes from.

**Impact:** False latency intuitions. Severity: P2.

### 10. [INCORRECT] HTTP/0.9 Request Format Has a Structural Error (Loopback vs. Real)

In loopback mode, the server handler waits for `\r\n` in the buffer to send a response. The actual HTTP/0.9 "Simple-Request" format from RFC 1945 Appendix D was:

```
GET /path SP CRLF
```

or even just `GET /path LF` (no CRLF required). The code sends `GET /hello.html\r\n` — which is correct. However, the educational framing does not mention that HTTP/0.9 had no CRLF requirement (many servers accepted bare LF) and that the spec predates CRLF normalization.

More importantly, in real mode when connecting to modern servers, HTTP/0.9-style requests (`GET /path\r\n`) will often be ignored, rejected, or met with a 400 Bad Request. The simulator shows this as a timeout, but doesn't explain why — missing the teaching moment.

**Impact:** Incomplete conceptual picture. Severity: P3.

---

### 11. [INCORRECT] SSE Real Mode Loses the `Last-Event-ID` Reconnect Mechanism

The SSE spec (WHATWG Streams) mandates:
1. On disconnect, the browser auto-reconnects after `retry:` interval
2. On reconnect, it sends `Last-Event-ID` header with the last received event ID
3. The server resumes the stream from that point

The recent git change removed this behavior entirely from the SSE route. The `SSERequest` interface no longer has `lastEventId`. The frontend's `LifecycleStep` type still has a `lastEventId` field (dead field). SSE reconnect is one of the most important educational concepts in this module — its absence makes the SSE simulator materially incomplete.

**Impact:** Core SSE concept is undemonstrated. Severity: P2.

### 12. [ARCHITECTURAL] Long Polling Real Mode Cannot Prove Long Polling Occurred

In long polling, the distinguishing characteristic is that the server **holds the connection open** until data is available. The real-mode implementation measures TTFB (time to first byte) and calls it "hold duration." But:

- A server that responds immediately (TTFB = 5ms) and a server that genuinely holds for 5s both produce the same visual output.
- There is no way to detect whether the server actually implemented long-polling semantics or just responded quickly.
- The "real mode" effectively just benchmarks HTTP response time and mislabels it as long polling.

**Impact:** Learners cannot distinguish long polling from normal HTTP in real mode. Severity: P2.

### 13. [ARCHITECTURAL] SSE Real Mode Uses `fetch()` — Cannot Show Protocol Layers

Unlike the HTTP lifecycle simulator (which uses raw `net`/`tls` sockets), SSE real mode uses:

```typescript
fetchRes = await fetch(url, { ... });
```

This means:
- DNS, TCP, and TLS are completely opaque — bundled into a single "connect" timing
- The `text/event-stream` wire format is never shown
- The `\n\n` event delimiter is processed internally by the SSE parser but not visualized
- The learner cannot see that SSE is just HTTP with a specific body format

The virtual mode correctly simulates discrete DNS/TCP/TLS/Request steps. The real mode collapses them. This inconsistency is confusing and means the real mode teaches less than the virtual mode.

**Impact:** Real mode is educationally inferior to virtual mode. Severity: P2.

### 14. [ARCHITECTURAL] HOL Blocking Demo Is Pure Simulation

The HOL blocking page (`/app/simulate/hol/page.tsx`) does not use actual HTTP/1.1 socket connections or HTTP/2 streams. It is a pure UI animation showing what HOL blocking would look like. There are no real network calls, no real TCP connections, no ALPN negotiation for HTTP/2.

This makes the HOL demo a visualization tool rather than a demonstration. A learner cannot observe HOL blocking actually occurring.

**Impact:** Claims "demonstrate HOL blocking" but demonstrates a drawing of HOL blocking. Severity: P2.

### 15. [ARCHITECTURAL] `globalThis.__simSessions` Is Not Horizontally Scalable

The step-mode session store uses in-process memory. In a multi-instance deployment (e.g., Vercel serverless functions), each invocation is a separate process. A session created in instance A cannot be accessed by instance B. Steps 1-6 would fail silently if routed to different instances.

The current implementation works for local dev and single-instance deployments only. This is not documented.

**Impact:** Silent failure in production multi-instance environments. Severity: P2.

---

### 16. [BUG] Response Buffer Cap Applies After Full Headers Are Received

```typescript
const responseBody = decodedBody.slice(0, 10240);
```

The 10KB cap only applies to the response body after the full raw response (headers + body) is buffered in memory. A server returning large headers (e.g., 2MB of `Set-Cookie` headers) will fully buffer into memory before the cap applies. There is no total-bytes-received guard.

**Impact:** Potential OOM on adversarial responses. Severity: P2.

---

### 17. [BUG] Content-Type Is Hardcoded to `application/json` for All Request Bodies

```typescript
if (body) {
  lines.push(`Content-Type: application/json`);
  lines.push(`Content-Length: ${Buffer.byteLength(body, "utf8")}`);
}
```

If a user provides a non-JSON body (form data, XML, plain text), the `Content-Type` header will be wrong. More importantly, user-provided `Content-Type` headers are silently dropped:

```typescript
if (lk === "content-type" && body) continue; // avoid duplicate; added below
```

The comment says "avoid duplicate" but it discards the user's value and replaces it with `application/json`.

**Impact:** Users cannot accurately simulate non-JSON POST requests. Severity: P3.

### 18. [BUG] Stage Route Has a Polling Race Condition for TTFB

```typescript
// Processing stage — polls for first byte:
const check = () => {
  if (session.firstByteAt !== undefined || session.responseFinished || Date.now() > deadline) {
    resolve();
  } else {
    setTimeout(check, 10);  // 10ms poll
  }
};
check();
```

The 10ms polling interval means TTFB measurement has ±10ms precision at best. If the server responds in 3ms, the reported TTFB will be 10-20ms. This is labelled as real timing but is actually polling-quantized. At 10ms granularity, the precision gap is educationally significant for fast local servers.

**Impact:** TTFB measurements are systematically inflated in step mode. Severity: P3.

## Limitations

### Protocol-Level

1. **HTTP/2 is not implemented.** ALPN negotiation, binary frame format, HPACK header compression, stream multiplexing — none of this exists. HTTP/2 is listed as "coming soon" but learners see it in the version comparison without experiencing it.

2. **HTTP/3 (QUIC) is conceptual only.** No QUIC implementation. Cannot be implemented in Node.js without native bindings or an external library.

3. **TLS is a black box.** The TLS handshake is measured as a single duration. Certificate chain validation, key exchange (ECDHE), cipher negotiation steps, and the specific TLS 1.3 vs 1.2 handshake round trips are not surfaced.

4. **TCP three-way handshake is opaque.** SYN, SYN-ACK, ACK packet timing is not shown. `net.createConnection` collapses all three into one async operation. Learners see "TCP: 14ms" but cannot see the SYN, SYN-ACK, ACK sequence.

5. **SSE `retry:` field is silently ignored.** The real-mode SSE parser skips `retry:` lines. Learners cannot see reconnect interval negotiation.

6. **WebSockets are not implemented.** Listed in the simulation hub as "coming soon."

7. **HTTP caching mechanics are absent.** ETags and `304 Not Modified` exist in the polling demo but the cache validation handshake (conditional GET) is not explained in protocol terms.

### Transport-Level

1. **IPv6 is not supported.** All DNS resolution forces IPv4 (`family: 4`). Learners cannot experience dual-stack, Happy Eyeballs, or IPv6-only network behavior.

2. **TCP congestion control, Nagle's algorithm, and receive window are invisible.** These fundamentally affect HTTP performance but are not surfaced.

3. **MTLS (mutual TLS) is not demonstrated.** Client certificates are never used.

4. **DNS over HTTPS (DoH) and DNS over TLS (DoT) are not demonstrated.**

5. **Packet loss, reordering, and jitter** are not simulated even in virtual mode. Learners cannot see how TCP handles them.

### Architectural

1. **No rate limiting.** Any user can hammer the real-mode endpoints and cause the server to open thousands of TCP connections.

2. **No concurrency limit per user session.** Multiple simultaneous `/api/simulate` calls are allowed.

3. **Session store uses no persistence.** Server restart loses all in-progress step-mode sessions.

4. **No request deduplication.** The same URL can be requested many times simultaneously.

5. **Next.js runtime edge vs. Node is not surfaced to learners.** The `export const runtime = "nodejs"` directive is critical but invisible.

### Product-Level

1. **Virtual mode does not explain *why* timings are what they are.** A learner sees "DNS: 12ms" but doesn't know if that's typical, fast, or slow.

2. **No comparison view** between virtual and real mode side-by-side.

3. **No "what if" scenarios.** What if DNS is slow? What if TLS negotiates TLS 1.2 instead of 1.3? Learners cannot experiment with degraded conditions.

4. **The HOL blocking demo does not let users configure stream sizes or counts** to see how the blocking relationship changes.

5. **Error scenarios are underdeveloped.** What does a connection refused look like? A TLS cert mismatch? A 5xx from a real server? These are important learning moments that have no dedicated flows.

## Conceptual Gaps

### Gap 1: Request "Sending" vs. Network Transit

The simulator shows "request sent: 0ms" which learners will interpret as "the request left instantly." In reality, 0ms reflects the OS buffer write time. The network transit of the request bytes is subsumed into TTFB and cannot be separately measured from the client side. This is not explained anywhere.

### Gap 2: What TLS Actually Protects (and Doesn't)

`rejectUnauthorized: false` is used in all TLS connections. This is correct for a simulator that needs to connect to any host, but learners are never told that in a real application, disabling cert validation means a MITM can impersonate any server. The simulator inadvertently normalizes dangerous TLS configuration.

### Gap 3: DNS Resolution Is OS-Dependent

The simulator uses `dns.lookup()` which delegates to the OS resolver. On different systems, the behavior changes fundamentally:
- macOS: mDNSResponder with local caching
- Linux: nscd (often not running) or systemd-resolved
- Docker: custom nameserver in `/etc/resolv.conf`

This is never explained. A learner who sees "DNS: 2ms" on macOS (cached) and "DNS: 45ms" on a Linux server may not understand why.

### Gap 4: Keep-Alive Is Not Free

The simulator shows keep-alive as a pure win (saved time = TCP + TLS). It doesn't show the tradeoffs:
- Persistent connections consume server resources per client
- Idle keep-alive connections block server connection pools
- HTTP/1.1 still serializes requests on one connection (no multiplexing)

### Gap 5: "Long Polling" Is a Server Contract, Not a Protocol

The simulator implies that long polling is detectable at the transport level (hold duration = server processing time). In reality, long polling is a server-side agreement — a server that holds connections for long polling looks identical on the wire to a slow server. This distinction is critical and is not made.

### Gap 6: SSE Is Unidirectional HTTP, Not a New Protocol

The SSE simulator's lifecycle shows DNS → TCP → TLS → Request as if SSE is a distinct protocol. It isn't — it's `GET /events HTTP/1.1` followed by a persistent response body with a specific text format. The simulator could reinforce this by showing the raw HTTP response headers (`Content-Type: text/event-stream`), which it does in virtual mode but not with the same clarity in real mode.

## Summary Verdict

The architecture is sound. The educational ambitions are high. The real TCP/TLS socket work is genuinely impressive for a Next.js project. However, the system has one production-breaking bug (SSE syntax error), three security holes (SSRF on three routes), and a pattern of "silent lies" — measurements and labels that are technically implemented but semantically misleading to a learner.

**The core mandate:** Fix correctness before adding features. A simulator that teaches wrong mental models is worse than no simulator.

Priority order:
1. Fix the syntax error (T1-1)
2. Patch all three SSRF gaps (T1-2, T1-3, T1-4)
3. Fix `savedMs` (T1-5)
4. Restore SSE reconnect (T2-1)
5. Fix request timing framing (T2-3)
6. Recalibrate virtual timings (T2-4)
7. Then build HTTP/2
