# Application Summary — 2026-03-25

> Snapshot of the full application state after the P0–P3 engineering review and fix pass.
> This document describes features, architecture, and limitations as they stand today.

---

## Overview

**Obsidian** is an interactive educational tool for learning HTTP internals. Every major
protocol concept — DNS resolution, TCP handshakes, TLS, HTTP/1.1 pipelining, long polling,
SSE streaming, Head-of-Line blocking — is taught through a combination of animated
simulations (virtual mode) and real network connections (real mode). The goal is to make
invisible wire-level behavior visible, measurable, and reproducible.

**Stack:** Next.js 16.2.0 · React 19.2.4 · TypeScript · Tailwind CSS 4 · Framer Motion
**Runtime:** Node.js (server routes use `export const runtime = "nodejs"` — raw sockets
require Node.js APIs unavailable in the Edge runtime)
**Deployment target:** Single-instance server (Vercel Pro, Railway, Fly.io, local dev).
Step-mode will not work reliably in multi-instance serverless environments.

## Modules

### 1. HTTP Lifecycle Simulator — `/simulate/http`

The flagship module. Teaches the full HTTP request/response lifecycle as six discrete stages,
each with its own measured duration.

#### Stages

| Stage | Virtual P50 | What it measures |
|-------|------------|-----------------|
| DNS | 35ms ± jitter | Name → IP resolution |
| TCP | 28ms | SYN → SYN-ACK → ACK |
| TLS | 45ms | ClientHello → Finished |
| Request | ~0ms | Bytes written to OS send buffer |
| Processing | 120ms | TTFB (request transit + server work + first byte transit, indistinguishable from client) |
| Response | 20ms | Body download |

Virtual jitter is applied to all stages except Request (which is near-zero and displayed
honestly). Durations are calibrated to real-world medians, not made-up round numbers.

#### Virtual mode

- Pre-built route library (GET /users, GET /users/:id, POST /users, DELETE /users/:id)
- Custom route editor: method, path, status code, delay, JSON response body
- Route path parameter substitution (`:id` → random integer)
- Configurable per-route delay simulates slow servers

#### Real mode

- Sends actual HTTP/HTTPS requests via raw `net`/`tls` sockets (not `fetch`), giving access
  to per-stage durations that the Fetch API does not expose
- TLS certificate inspection: subject, issuer, valid dates, fingerprint, cipher, version
- Full request/response header and body inspection
- Keep-alive demo: second request on the same socket skips DNS + TCP + TLS, showing
  `savedMs` = tcp + tls durations
- SSRF protection on every DNS resolution
- Presets: JSONPlaceholder, GitHub Zen, httpbin.org, localhost

#### Simulation modes

- **Auto:** stages play sequentially with delays
- **Step:** user advances each stage manually; socket is held between clicks (session store)
- **Timeout:** configurable (default 30s)

#### Key frontend components

| Component | Purpose |
|-----------|---------|
| `TopBar` | Mode toggle, method selector, URL input, run/stop |
| `LeftPanel` | Route editor and request builder (virtual) / URL + headers (real) |
| `LifecyclePanel` | Color-coded stage timeline with animated fill bars |
| `StageCard` | Per-stage detail: duration, description, real vs virtual label |
| `ResponsePanel` | Status code, headers, body with JSON tree viewer |
| `InspectOverlay` | Full-screen modal for certificate / header deep-dive |

### 2. Short Polling Simulator — `/simulate/polling`

Teaches the waste inherent in fixed-interval polling: most requests return nothing new.

#### Virtual mode

- Three events pre-scheduled (Order filled @4s, Message @9s, Alert @13s)
- Configurable poll interval
- Responses alternate between `200 OK` (with event data) and `304 Not Modified` (no change)
- ETag lifecycle: server issues an ETag on 200; client sends `If-None-Match` on subsequent
  requests; server returns 304 when data has not changed

#### Real mode

- Polls any URL at the configured interval
- Compares successive response bodies to detect actual data change
- Shows real round-trip durations and which rounds carried new data

#### Key metrics

- Total rounds vs data-bearing rounds
- Wasted bandwidth percentage
- Average round-trip time

### 3. Long Polling Simulator — `/simulate/long-poll`

Teaches the efficiency of server-held connections: the server waits until it has something
to say before responding. Fewer total requests for the same event delivery.

#### Per-round phases

1. **Connect** — DNS + TCP (simulated ~30–50ms)
2. **Hold** — server waits for an event (tick indicators every 400ms)
3. **Respond** — event delivered (~20ms)

#### Virtual mode

- Three events (Order filled @6s, Message @14s, Alert @22s)
- Each round holds until its event fires or `timeoutMs` is exceeded

#### Real mode

- Makes real HTTP requests to provided URL (e.g., httpbin.org/delay/N)
- Measures actual hold and response latency
- Honest caveat displayed: TTFB from a long-poll server and a slow server are
  indistinguishable on the wire

#### Analysis panel

- Side-by-side comparison of long-poll rounds vs equivalent short-poll rounds
- Total time, total bytes, connections opened

### 4. Server-Sent Events (SSE) Simulator — `/simulate/sse`

Teaches SSE: a single persistent HTTP connection over which the server pushes multiple
events at arbitrary times. Also demonstrates the `Last-Event-ID` reconnect mechanism.

#### Connection lifecycle (virtual mode — granular)

DNS → TCP → TLS → HTTP Request → Response Headers → Stream Open → Events → Close

Each step is timed individually and displayed in a lifecycle timeline.

#### Virtual mode

| Event | ID | Delay | Type |
|-------|----|-------|------|
| 1 | sse1 | 3s | notification |
| 2 | sse2 | 7s | notification |
| 3 | sse3 | 11s | alert |
| 4 | sse4 | 16s | notification |
| 5 | sse5 | 21s | close |

- Full SSE wire format emitted (`id:`, `event:`, `data:` lines)
- `Last-Event-ID` reconnect: on reconnect, server emits a `reconnect_resume` event showing
  how many events were skipped, then resumes from the next undelivered event

#### Real mode

- Proxies to any SSE endpoint
- Detects `Content-Type: text/event-stream` to distinguish true SSE from plain HTTP
- Parses `id:`, `event:`, `data:` fields from the wire stream
- Forwards `Last-Event-ID` header to upstream on reconnect
- Callout: real mode cannot separate DNS/TCP/TLS/request timing (Node.js `fetch()` bundles
  them); links to the HTTP Lifecycle simulator for per-layer breakdown
- Emits actual response headers into the headers panel with educational annotations

#### Key frontend components

| Component | Purpose |
|-----------|---------|
| `SSEPanel` | Live event stream, elapsed time, disconnect button |
| `LifecycleTimeline` | DNS → TCP → TLS → headers → stream_open phase display |
| `HeadersPanel` | Response headers with explanations (why `no-cache`, `chunked`, etc.) |
| `SSEEventInspector` | Selected event details (ID, type, raw data, received timestamp) |
| `SSEAnalysis` | Session summary: total events, reconnect count, event list |

### 5. HOL Blocking Simulator — `/simulate/hol`

Teaches Head-of-Line blocking at the HTTP application layer: one slow request on an
HTTP/1.1 connection blocks every request queued behind it. HTTP/2 eliminates this by
multiplexing all streams independently.

#### Animated simulation

Six realistic page resources (vendor.js 3s, app.js 1.2s, api.json 1s, hero.jpg 0.8s,
style.css 0.6s, thumb.jpg 0.5s) distributed across two HTTP/1.1 connections. Gantt bars
animate at 1.5× speed, showing:

- Connection 1: vendor → app → api (vendor blocks app and api for 3s)
- Connection 2: style → hero → thumb (unaffected, runs in parallel)
- HTTP/1.1 total: **5.2s**
- HTTP/2 total: **3.0s** (all streams start at t=0)

#### Live TCP demo

Spins up a real `net.createServer()` loopback server with three endpoints:

- `/slow` — 2000ms server delay
- `/fast1` — instant
- `/fast2` — instant

**Phase 1 (HOL):** One TCP connection. All three requests are pipelined (sent before any
response arrives). The server processes them in order; `/fast1` and `/fast2` cannot be
delivered until `/slow` finishes. Real response timestamps are streamed back and rendered
as timeline bars.

**Phase 2 (Parallel):** Three TCP connections, one request each, all launched
simultaneously. `/fast1` and `/fast2` arrive in single-digit milliseconds; `/slow` takes
its full 2s.

The insight footer shows exact milliseconds: e.g., `/fast1` waited 2003ms on one connection
vs 4ms in parallel — 1999ms wasted on HOL blocking. These are actual socket timings, not
animation.

### 6. HTTP Versions Playground — `/simulate/http/versions`

Teaches the evolution of HTTP from 0.9 to 1.1 through wire-level differences.

| Version | Key wire differences |
|---------|---------------------|
| 0.9 | `GET /path\r\n` only — no headers, no status line in response |
| 1.0 | Full request line + headers, but no `Host` header requirement, no keep-alive by default |
| 1.1 | `Host` required, chunked transfer, persistent connections by default |

#### Features

- Loopback mode: server responds in the exact format of the requested version
- Real mode: sends version-correct requests to actual servers; detects and explains
  modern server rejection of HTTP/0.9 (most servers return `HTTP/1.1 400 Bad Request`)
- Educational callouts for two HTTP/0.9 failure cases:
  1. Server rejected with a proper HTTP/1.x response (shows the irony: the rejection
     itself violates the protocol it's rejecting)
  2. Server silently closed with no response

## Backend API Routes

### `/api/simulate` — Full request (POST, SSE response)

Executes a complete HTTP/HTTPS request end-to-end and streams stage events back.

```typescript
// Request body
{
  method: string,
  url: string,
  headers?: Record<string, string>,
  body?: string,
  keepAlive?: boolean
}

// Event stream shape (one JSON object per data: line)
{ type: "stage",    stage: "dns"|"tcp"|"tls"|"request"|"processing"|"response", ... }
{ type: "response", status, headers, body, ... }
{ type: "error",    message }
{ type: "done" }
```

- Uses raw `net`/`tls` sockets for sub-stage timing visibility
- `Content-Type` defaults to `application/json` only if user did not provide one
- `savedMs` on keep-alive = tcp + tls duration (correct, both hoisted before the if-block)
- SSRF check applied to resolved IP before any TCP connection

### `/api/simulate/stage` — Step-based request (POST, JSON response)

Executes one stage at a time. Socket is kept alive in a session store between clicks.

```typescript
// Request body
{
  stage: "dns" | "tcp" | "tls" | "request" | "processing" | "response",
  sessionId?: string,
  url?: string,
  method?: string,
  headers?: Record<string, string>,
  body?: string
}
```

- Sessions stored in `globalThis.__simSessions` (process memory, 60s TTL)
- TTFB measurement: `session._firstByteResolve` callback fires the instant the socket
  data event fires — no polling lag, sub-millisecond precision
- SSRF check on DNS stage

### `/api/long-poll` — Long-poll driver (POST, SSE response)

```typescript
{ mode: "virtual"|"real", timeoutMs: number, sessionStartedAt?: number,
  firedEventIds?: string[], url?: string }
```

- Virtual: three pre-scheduled events, hold ticks every 400ms
- Real: makes HTTP GET to user URL, measures actual hold time
- SSRF protection in real mode

### `/api/sse` — SSE driver (POST, SSE response)

```typescript
{ mode: "virtual"|"real", sessionStartedAt?: number, url?: string,
  timeoutMs?: number, lastEventId?: string }
```

- Virtual: five events with IDs; reconnect skips already-delivered events via `slice(startIdx)`
- Real: proxies upstream SSE stream; forwards `Last-Event-ID` header; detects non-SSE endpoints
- Note: Vercel Hobby has a 10s function limit; virtual events run for 21s (use Pro or local dev)

### `/api/hol` — HOL live demo (GET, SSE response)

No input. Internally:

1. Starts a `net.createServer()` on a random loopback port
2. Phase 1: opens one TCP socket, pipelines GET /slow + GET /fast1 + GET /fast2, records arrival times
3. Phase 2: opens three TCP sockets simultaneously, one request each, records arrival times
4. Closes server, emits `done`

Events streamed: `server_ready`, `phase`, `request_sent`, `response_received`, `done`, `error`

### `/api/http-versions` — HTTP version runner (POST, JSON response)

```typescript
{ version: "0.9"|"1.0"|"1.1", target?: string, loopback?: boolean }
```

- Loopback mode: internal server responds in the exact wire format of the requested version
- Real mode: builds version-correct raw request string, sends via `net.Socket`, returns raw response
- SSRF protection in real mode

## Shared Libraries

### `lib/ip-guard.ts`

```typescript
export function isBlockedIp(ip: string): boolean
```

Blocks connections to:
- **IPv4:** 0.0.0.0/8, 10.0.0.0/8, 127.0.0.0/8, 100.64.0.0/10 (CGNAT), 169.254.0.0/16
  (link-local / AWS metadata), 172.16.0.0/12, 192.0.0.0/24, 192.168.0.0/16,
  198.18.0.0/15, 198.51.100.0/24, 203.0.113.0/24, 224.0.0.0/4+
- **IPv6:** `::1`, `::`, `fe80::/10`, `fc00::/7` (ULA), `ff00::/8` (multicast)

Called by every real-mode API route after DNS resolution, before opening a TCP socket.

### `lib/sim-sessions.ts`

Persistent socket state for step-mode. Sessions are stored in `globalThis.__simSessions`
(a `Map<string, SimSession>`) to survive Next.js hot-reload in development.

```typescript
interface SimSession {
  id, createdAt, ip, hostname, port, isHttps,
  rawSocket, socket,          // live OS file descriptors
  tlsVersion, tlsCipher, tlsCert,
  responseChunks, firstByteAt, responseFinished, responseFinishedAt,
  requestSentAt, requestRaw,
  _timer,                     // 60s TTL expiry
  _firstByteResolve           // callback — fires when first response byte arrives
}
```

Functions: `createSession`, `getSession`, `touchSession`, `destroySession`

**Hard limitation:** Sessions hold live TCP/TLS sockets. Sockets are OS file descriptors
tied to the process that opened them. They cannot be serialized, put in Redis, or shared
across processes. In any multi-instance deployment, step requests routed to a different
instance than the one that created the session will fail with "session not found". A
`console.warn` fires once on Vercel to surface this immediately.

## Known Limitations

| # | Area | Description | Severity |
|---|------|-------------|----------|
| 1 | Session store | `globalThis.__simSessions` is per-process; step-mode fails silently in multi-instance deployments (Vercel, scaled containers) | P2 |
| 2 | TLS validation | All real-mode connections use `rejectUnauthorized: false`. Self-signed certs are accepted. Documented in code; not yet surfaced in the UI as a learner callout. | P2 |
| 3 | Vercel function timeout | Long-poll and SSE virtual modes run for 21–30s. Vercel Hobby has a 10s limit. Documented in route files. | P3 |
| 4 | Chunked encoding | `sendHttpRequestKeepAlive.trySettle()` checks `body.includes("\r\n0\r\n\r\n")` to detect the terminal chunk. This is a string search on the concatenated body, which can false-positive if a response body happens to contain those bytes. | P2 |
| 5 | Real mode request body | No validation prevents `\r\n` injection in user-supplied header values or request body, though the educational context makes this low-risk. | P3 |
| 6 | HOL loopback port | Creates an ephemeral `net.Server` on a random port per request. Port is released after the demo completes. Under high concurrency, port exhaustion is theoretically possible. | P3 |
| 7 | HTTP/2 not wire-observable | HTTP/2 uses binary framing over TLS; it cannot be replicated with raw `net`/`tls` sockets the way HTTP/1.x can. The HTTP/2 panel explains this honestly but does not attempt a real connection. | Architectural |
| 8 | HTTP/3 / QUIC | No implementation. QUIC is UDP-based and requires a QUIC library. Shown as "coming soon" in the versions playground. | Architectural |
| 9 | Short polling | No dedicated API route — the polling module makes direct `fetch()` calls from the server component rather than using a raw socket driver like the other modules. | P3 |

## Security Posture

| Control | Status |
|---------|--------|
| SSRF — IP blocklist | Applied in all real-mode routes after DNS resolution |
| SSRF — DNS rebinding | Not protected. A hostname could resolve to a public IP at DNS time, then rebind to a private IP before TCP connects. Acceptable risk for a local-dev educational tool. |
| TLS cert validation | Disabled (`rejectUnauthorized: false`) — intentional for educational access to servers with self-signed certs |
| Request header injection | Not sanitized — headers passed through as-is |
| Response size cap | `body.slice(0, 10240)` applied in keep-alive and non-SSE response paths; not uniformly applied |
